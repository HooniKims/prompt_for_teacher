# 모듈화와 로컬 메모리 설계

## 배경

이 문서는 `handoff.md`에 정리된 현재 웹앱을 다음 단계로 고치기 위한 설계다.
제품 방향은 `docs/prd/0001-teacher-prompt-guide-prd.md`를 따르고, 용어는 `CONTEXT.md`를 따른다.
여기서는 PRD 내용을 반복하지 않고, 이번에 바꿀 구조만 다룬다.

현재 `webapp/main.js`는 너무 많은 일을 한 파일에서 처리한다.

- 10단계 질문 데이터
- 현재 대화 상태
- 최종 프롬프트 생성
- 참고 안내 생성
- 수정 요청 반영
- `localStorage` 저장과 불러오기
- 화면 렌더링
- 버튼과 입력창 이벤트 연결

그래서 작은 기능을 고치려고 해도 관련 없는 코드까지 같이 읽어야 한다.
앞으로 LM Studio 연동이나 메모리 기능이 들어가면 더 복잡해질 가능성이 크다.

## 목표

웹앱을 기능별 모듈로 나누고, 사용자가 켜고 끌 수 있는 로컬 요약 메모리를 추가한다.
구조를 나누는 이유는 단순한 정리가 아니다.
교사가 도우미와 대화하듯 생각을 정리하고, 마지막에는 바로 실행할 수 있는 프롬프트를 얻도록 만들기 위해서다.

앱은 다음을 만족해야 한다.

- 교사에게 답을 대신 정해주지 않고, 교사가 스스로 판단하도록 돕는다.
- 최종 프롬프트는 다른 AI 도구에 바로 붙여 넣어 실행할 수 있을 만큼 구체적이어야 한다.
- 최종 프롬프트는 AI가 해석하기 쉬우면서도 사람이 읽고 수정하기 쉬워야 한다.
- 진행 중인 채팅 기록과 완성된 세션을 사용자의 브라우저 `localStorage`에 저장한다.
- 사용자가 페이지를 새로고침해도 진행 중이던 대화를 이어갈 수 있다.
- 완료된 작업에서 가벼운 요약 메모리를 만든다.
- 다음 작업에서 이 메모리를 선택적으로 참고한다.
- 사용자가 메모리 사용을 끌 수 있다.
- 모든 데이터는 사용자의 브라우저 안에만 둔다.
- 백엔드, 로그인, 원격 동기화는 만들지 않는다.

## 추천 방식

현재처럼 정적 HTML, CSS, 바닐라 JavaScript를 유지하되, JavaScript를 ES 모듈로 나눈다.

이 방식이 맞는 이유는 다음과 같다.

- 현재 앱은 정적 mock이다.
- 빌드 시스템이 없다.
- 브라우저가 ES 모듈을 바로 불러올 수 있다.
- 이번 요구사항은 로컬 저장만으로 충분하다.
- LM Studio 연동은 나중에 별도 모듈로 붙일 수 있다.

## 모듈 구조

`webapp/index.html`

- 기존 구조를 유지한다.
- `<script src="./main.js"></script>`는 `<script type="module" src="./main.js"></script>`로 바꾼다.
- 왼쪽 히스토리 패널에 메모리 상태와 조작 버튼이 들어갈 자리만 추가한다.

`webapp/main.js`

- 앱 시작점이다.
- 필요한 모듈을 불러온다.
- 초기 상태를 만든다.
- 저장된 진행 중 초안이 있으면 불러와 이어서 시작한다.
- UI 콜백과 상태 변경 함수를 연결한다.
- 첫 화면 렌더링을 실행한다.
- 프롬프트 본문, 저장소 직렬화, DOM 생성 세부 코드는 두지 않는다.

`webapp/modules/steps.js`

- 10단계 질문 흐름을 담당한다.
- 단계 목록과 단계 조회 함수를 내보낸다.
- 교사에게 보이는 질문 문구를 상태 제어 코드와 분리한다.

`webapp/modules/state.js`

- 현재 세션 상태를 만들고 갱신한다.
- 다음 전환을 처리한다.
  - 첫 요청 입력
  - 단계별 답변 입력
  - 10단계 완료
  - 수정 요청 입력
  - 저장된 세션 불러오기
  - 진행 중 초안 복원
- `localStorage`를 직접 읽거나 쓰지 않는다.
- DOM을 직접 만지지 않는다.

`webapp/modules/promptBuilder.js`

- 정리된 세션 상태를 받아 복사 가능한 최종 프롬프트를 만든다.
- 메모리 사용이 켜져 있으면 짧은 메모리 문맥을 함께 받는다.
- 메모리는 "확정 사실"이 아니라 "이전 작업에서 나온 참고 경향"으로만 표현한다.
- 최종 프롬프트는 사람이 읽는 문서처럼 구조가 보여야 하고, AI가 수행할 작업도 분명해야 한다.
- 역할, 상황, 목표, 사용 가능한 정보, 작업 지시, 제약 조건, 출력 형식, 품질 기준, 정보 부족 시 행동을 빠뜨리지 않는다.

`webapp/modules/referenceNotes.js`

- 위험 신호를 받아 참고 안내를 만든다.
- 개인정보와 학습지원 소프트웨어 관련 안내를 `promptBuilder.js`와 분리한다.
- 최종 프롬프트와 참고 안내가 섞이지 않게 한다.

`webapp/modules/revision.js`

- 사용자의 자연어 수정 요청을 현재 최종 프롬프트에 반영한다.
- 다음 두 값을 반환한다.
  - 수정된 프롬프트
  - 짧은 변경 요약

`webapp/modules/storage.js`

- 모든 `localStorage` 키와 저장 형식을 담당한다.
- 다음 기능을 제공한다.
  - 저장된 세션 읽기
  - 완료된 세션 저장하기
  - 진행 중 초안 읽기
  - 진행 중 초안 쓰기
  - 진행 중 초안 지우기
  - 저장된 세션 지우기
  - 메모리 읽기
  - 메모리 쓰기
  - 메모리 지우기
  - 설정 읽기
  - 설정 쓰기
- 저장된 JSON이 깨져 있어도 앱이 멈추지 않게 기본값을 반환한다.

`webapp/modules/memory.js`

- 완료된 세션에서 가벼운 메모리를 만든다.
- 다음 작업에 쓸 메모리 항목을 고른다.
- 전체 채팅 전문을 메모리로 저장하지 않는다.
- `localStorage`에 직접 접근하지 않는다. 일반 JavaScript 객체를 입력받고 일반 JavaScript 객체를 반환한다.

`webapp/modules/ui.js`

- DOM 조회, 화면 렌더링, 이벤트 연결을 담당한다.
- 다음 화면 요소를 그린다.
  - 채팅 메시지
  - A/B/C/D 선택지
  - 단계 진행 기록
  - 최종 프롬프트와 참고 안내
  - 저장된 세션 목록
  - 메모리 켜기/끄기
  - 토스트 메시지
- 상태를 직접 바꾸지 않고, `main.js`에서 받은 콜백을 호출한다.

## 대화 원칙

이 앱의 가이드는 교사를 검사하거나 대신 판단하는 역할이 아니다.
교사가 생각을 정리하도록 옆에서 묻고, 선택지를 제안하고, 판단 지점을 드러내는 도우미 역할을 한다.

대화는 다음 원칙을 따른다.

- 한 번에 하나의 질문만 한다.
- 질문은 교사가 실제로 답할 수 있는 쉬운 말로 쓴다.
- 선택지는 빠른 응답을 돕기 위한 것이며, 직접 답변도 항상 허용한다.
- AI가 대신 결정하면 안 되는 부분은 **교사 판단 지점**으로 분리한다.
- 안전 확인은 겁주는 문구가 아니라 "무엇을 그대로 넣으면 위험한지"를 알려주는 방식으로 묻는다.
- 사용자가 모호하게 답하면 단정하지 말고 확인 질문을 남긴다.
- 메모리는 교사의 이전 경향을 참고하는 용도이지, 현재 요청을 덮어쓰는 근거가 아니다.

## 최종 프롬프트 품질 기준

최종 프롬프트는 교사와 가이드가 함께 만든 실행 지시문이어야 한다.
읽는 사람은 이 프롬프트가 왜 이렇게 구성되었는지 이해할 수 있어야 하고, AI 도구는 바로 작업을 수행할 수 있어야 한다.

최종 프롬프트는 다음 구조를 기본으로 한다.

- `[역할]`: AI가 어떤 협력자로 행동해야 하는지
- `[상황]`: 교사의 초기 요청, 산출물, 사용 장면, 교육 및 업무 맥락
- `[목표]`: 이번 작업에서 좋아져야 하는 부분
- `[교사 판단 지점]`: AI가 단정하지 말고 교사에게 선택권을 남겨야 하는 부분
- `[사용 가능한 정보]`: 참고 자료, 안전 확인, 외부 도구 사용 여부
- `[작업 지시]`: AI가 실제로 해야 할 일
- `[반드시 지킬 조건]`: 개인정보, 외부 서비스, 교사 주도성, 말투 조건
- `[출력 형식]`: 표, 단계별 계획, 문장 초안, 체크리스트 등
- `[품질 기준]`: 현장에서 바로 쓸 수 있는지 판단할 기준
- `[정보가 부족할 때]`: 추측하지 말고 확인 질문을 먼저 하라는 지시

좋은 최종 프롬프트의 기준은 다음과 같다.

- 교사가 읽어도 문맥과 의도가 이해된다.
- AI가 읽어도 역할, 작업, 제약, 출력 형식이 분명하다.
- 개인정보나 외부 서비스 관련 위험 조건이 프롬프트 안에서 빠지지 않는다.
- "좋게 만들어줘" 같은 막연한 지시가 아니라, 산출물과 품질 기준이 구체적이다.
- 교사의 판단을 대신하지 않고, 필요한 경우 선택지와 장단점을 제시하게 한다.
- 수정 요청을 반영한 뒤에도 전체 프롬프트가 다시 깨끗한 한 벌의 문서로 정리된다.

`teacherPromptGuideSessions`

현재 앱이 이미 쓰고 있는 이전 저장 키다.

이관이 끝나면 이전 키는 지워도 된다.
이관에 실패하면 기존 값은 남겨두고 새 구조는 빈 기본값으로 시작한다.

## 메모리 동작

메모리는 10단계 흐름이 끝난 뒤에만 만든다.
중간에 작성하다 만 대화는 메모리로 삼지 않는다.
진행 중 초안은 이어서 작업하기 위한 저장값일 뿐, 다음 작업에 참고할 메모리는 아니다.

첫 구현에서는 기존 답변에서 규칙 기반으로만 메모리를 뽑는다.
AI 요약은 쓰지 않는다.

추출 대상은 다음으로 제한한다.

- `context`: 교육 및 업무 맥락
- `output`: 선호하는 산출물
- `useScene`: 사용 장면
- `quality`: 품질 기준
- 개인정보 또는 외부 서비스 관련 위험 선택지

메모리는 `memoryEnabled`가 `true`일 때만 사용한다.

메모리 사용이 켜져 있으면:

- 첫 안내 메시지에서 로컬 메모리가 켜져 있음을 짧게 보여준다.
- `promptBuilder`에 짧은 메모리 문맥을 넘긴다.
- 최종 프롬프트에는 메모리를 이전 작업의 경향으로 넣는다.

최종 프롬프트에 들어갈 수 있는 문구 예시는 다음과 같다.

```text
[이전 작업 참고]
아래 내용은 이 브라우저에 저장된 이전 작업 요약이다. 현재 요청과 맞지 않으면 따르지 말고, 필요한 경우 사용자에게 확인 질문을 하라.
- 중학교 수업 맥락을 자주 사용함
- 결과물은 표 형식을 선호하는 경우가 많음
```

메모리 사용이 꺼져 있으면:

- 프롬프트 생성에 메모리를 넘기지 않는다.
- 세션 기록은 계속 저장할 수 있다.
- 이미 저장된 메모리는 사용하지 않는다.
- 사용자가 `CLEAR MEMORY`를 누르기 전까지 기존 메모리 자체는 남아 있다.

## 화면 변경

왼쪽 히스토리 패널의 저장된 초안 아래에 작은 메모리 영역을 추가한다.

필요한 조작은 다음 세 가지다.

- 메모리 켜기/끄기
- 메모리 삭제
- 현재 메모리 사용 상태 표시

화면 문구는 길게 설명하지 않는다.
짧은 라벨만 둔다.

- `MEMORY ON`
- `MEMORY OFF`
- `CLEAR MEMORY`

저장된 세션은 계속 클릭해서 불러올 수 있어야 한다.
세션을 불러오면 최소한 다음은 복원한다.

- 최종 프롬프트 미리보기
- 참고 안내
- 저장 당시의 핵심 세션 정보

진행 중 초안이 있으면 페이지 진입 시 바로 이어서 작업할 수 있게 복원한다.
복원 대상은 현재 단계, 채팅 메시지, 답변, 최종 프롬프트, 참고 안내다.
사용자가 새로 시작하고 싶을 때를 위해 현재 초안을 지우는 동작도 필요하다.

## 디자인 기준

화면 구현은 `/design` 폴더의 기존 디자인 기준을 따른다.

참고 파일은 다음과 같다.

- `design/DESIGN.md`: 전체 스타일 방향
- `design/variables.css`: 실제 웹앱에서 가져다 쓰는 CSS 변수
- `design/theme.css`: 추가 테마 기준
- `design/tokens.json`: 색상, 간격, 타이포그래피 토큰

이번 변경에서 새 UI를 만들 때 지킬 기준은 다음과 같다.

- 기존 dark monochrome 분위기를 유지한다.
- `styles.css`는 계속 `@import "../design/variables.css";`를 사용한다.
- 본문과 UI 폰트는 프로젝트 폴더에 내려받은 Paperlogy 웹폰트를 우선 사용한다.
- 기본 자간은 `-0.02em`, 기본 줄간격은 `1.3` 정도로 맞춘다.
- 새 색상 값을 임의로 많이 추가하지 않는다.
- 강조가 필요하면 기존 subtle orange 계열만 제한적으로 사용한다.
- 버튼, 배지, 카드 반경은 현재 화면의 8px 또는 pill 스타일과 맞춘다.
- 메모리 영역은 새 큰 패널이 아니라 왼쪽 히스토리 패널 안의 작은 설정 블록으로 둔다.
- `MEMORY ON`, `MEMORY OFF`, `CLEAR MEMORY` 같은 조작 문구는 기존 `text-button`, `status-pill`, `small-badge` 스타일을 재사용한다.
- 설명 문구를 길게 넣지 않는다. 이 앱은 교사가 작업에 집중하는 도구이므로, 기능 설명보다 현재 상태와 조작이 바로 보여야 한다.

즉, 이번 작업은 새 디자인을 만드는 것이 아니다.
기존 디자인 시스템 안에서 메모리 기능을 자연스럽게 붙이는 작업이다.

## 오류 처리

`storage.js`는 다음 상황을 처리해야 한다.

- `localStorage`를 사용할 수 없음
- 저장된 JSON이 깨짐
- 저장된 값의 형태가 예상과 다름
- 브라우저 저장 용량 초과
- 이전 저장 키에서 새 저장 키로 이관 실패

기대 동작은 다음과 같다.

- 현재 진행 중인 대화는 가능한 한 계속 사용할 수 있어야 한다.
- 저장 실패는 토스트로 알려준다.
- 읽기 실패는 빈 기본값을 반환한다.
- 이관 실패는 기존 값을 지우지 않는다.
- 이 앱이 쓰는 키 외의 다른 로컬 저장소 값은 건드리지 않는다.

## 테스트 전략

아직 테스트 환경이 없으므로, 첫 구현에서는 브라우저 없이 확인 가능한 순수 모듈부터 작게 테스트한다.
별도 빌드 도구를 추가하지 않고 Node.js의 기본 테스트 실행기인 `node --test`를 우선 사용한다.

우선 확인할 동작은 다음과 같다.

- `memory.js`가 완료된 세션에서 예상한 메모리를 뽑는지 확인한다.
- 메모리 사용이 꺼져 있으면 프롬프트 문맥이 비는지 확인한다.
- `storage.js`가 깨진 JSON을 만나도 기본값을 반환하는지 확인한다.
- `storage.js`가 기존 `teacherPromptGuideSessions` 값을 새 세션 구조로 이관하는지 확인한다.
- `state.js`가 진행 중 초안을 복원해 같은 단계에서 이어갈 수 있는지 확인한다.
- `promptBuilder.js`가 메모리 문맥을 받은 경우에만 `[이전 작업 참고]`를 넣는지 확인한다.
- `promptBuilder.js`가 역할, 상황, 목표, 교사 판단 지점, 사용 가능한 정보, 작업 지시, 조건, 출력 형식, 품질 기준, 정보 부족 시 행동을 모두 포함하는지 확인한다.
- `revision.js`가 수정 요청 뒤에도 최종 프롬프트를 조각난 덧붙임이 아니라 읽기 쉬운 한 벌의 프롬프트로 유지하는지 확인한다.
- 참고 안내가 최종 프롬프트 안에 섞이지 않는지 확인한다.

수동 브라우저 확인은 다음 흐름으로 한다.

- 10단계 흐름을 끝낸다.
- 저장된 세션 목록에 새 항목이 생기는지 확인한다.
- 페이지를 새로고침해도 저장된 세션이 남아 있는지 확인한다.
- 10단계 중간에 페이지를 새로고침해도 진행 중 대화가 이어지는지 확인한다.
- 메모리 켜기/끄기 상태가 새로고침 뒤에도 유지되는지 확인한다.
- 메모리를 끄면 다음 최종 프롬프트에 이전 작업 참고가 빠지는지 확인한다.
- 메모리를 삭제하면 저장된 메모리 항목이 사라지는지 확인한다.
- 완성된 최종 프롬프트를 읽었을 때 교사도 구조를 이해할 수 있고, 다른 AI 도구에 바로 붙여 넣을 수 있는지 확인한다.
